# Incident Management

## Overview

The Incident Management module supports **ISO/IEC 27001:2022 Annex A controls A.5.24–A.5.28** and links to **Clause 10.2** (nonconformity and corrective action) when incidents reveal systemic gaps.

Organizations must plan for, detect, assess, respond to, learn from, and preserve evidence for information security incidents. IdaraOS provides a structured incident register with ownership, severity classification, response lifecycle tracking, evidence linking, and version history for published records.

## ISO 27001:2022 Mapping

| Control / Clause | Requirement | IdaraOS Coverage |
|------------------|-------------|------------------|
| **A.5.24** Planning & preparation | Define incident processes, roles, communication, and response procedures | Incident register, owner assignment, severity (P1–P4), status workflow |
| **A.5.25** Event assessment | Assess events and decide if they are incidents | Classification: event vs incident; triage status |
| **A.5.26** Incident response | Containment, eradication, recovery per documented procedures | Response fields: containment, eradication, recovery actions; timestamps |
| **A.5.27** Lessons learned | Use incident knowledge to improve controls | Root cause analysis, lessons learned, linked evidence |
| **A.5.28** Evidence collection | Identify, collect, acquire, preserve evidence | Multiple linked evidence items from Evidence Store |
| **Clause 10.2** Improvement | Address nonconformities arising from incidents | Link to Nonconformity & Corrective Action module |

## Module Location

**Security → Incident Management** (`/security/incidents`)

## Key Features

### Incident Register

Create, view, update, and delete incidents scoped to your organization. Each incident has a unique `incidentId` (e.g. `INC-001`).

### Owner Assignment

Assign an incident owner from **active platform users** (not directory/person records). The owner is accountable for coordination and closure.

### Classification & Severity

| Field | Values | ISO Alignment |
|-------|--------|---------------|
| Classification | Event, Incident | A.5.25 — event vs incident decision |
| Severity | P1 (Critical) – P4 (Low) | A.5.24 — priority levels and escalation |
| Status | Draft → Reported → Triaging → Responding → Resolved → Closed | A.5.26 — response lifecycle |

### Publication & Version History

When an incident is **published**, subsequent edits create a **version snapshot** (same pattern as the Documentation module). This preserves retained information for audits and demonstrates change control over incident records.

- **Draft** — editable without versioning
- **Published** — edits bump `currentVersion` and store prior state in `security_incident_versions`

### Evidence Linking

Attach multiple evidence items from the Evidence Store. Supports A.5.28 chain-of-custody and audit readiness.

### Response Documentation

Capture impact, containment, eradication, recovery, root cause analysis, and lessons learned — aligned with A.5.26 and A.5.27.
